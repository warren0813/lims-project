## 一、流程定義

**【Q】** 題目中的「送件」與「接件」是否都是指樣品的送出與接收？委託單的「送出」和樣品的「送樣」是不同的事情嗎？

**【A】** 對。你說廠區使用者在系統上把委託單送出，送件那邊指的是樣品，實體的樣品。

**【Q】** 是否需要模擬送件過程，還是送件後一定會到達？如果樣品沒送到怎麼辦？

**【A】** 好問題。沒到的情況就是你那一張委託單就一直卡著嘛，等不到實驗室的人去做簽收。所以實務上是有可能的。

## 二、角色

**【Q】** 簡報中的實驗室人員是否分成實驗室人員與實驗室主管兩種角色？還是簽核、分派都由實驗室人員處理？

**【A】** 對，他就是要確保那個帳是對的嘛。實務上會有分不同職責，不過這邊就用實驗室的一般人員跟實驗室主管這兩種去規劃就好了。

**【Q】** 廠區使用者是否有很多個？

**【A】** 會會，會有很多個。

## 三、內容欄位設計

**【Q】** 委託單、樣品、recipe、上下貨紀錄、實驗結果這些內容，都是由我們自行設計，還是有特別規定？

**【A】** 自己設定，合理就可以了。

**【Q】** 樣品的 ID 要怎麼設計？用 ABCD 這種代號可以嗎？

**【A】** 你指的 ABCD 可能是某一種編號的樣子。我覺得描述具體一點好了，樣品你可以說就是一片 wafer。wafer 就會有 wafer ID 嘛，就這樣子去規劃，不然 ABCD 看面上看起來也很奇怪。

**【Q】** 那樣品還有其他需要記錄的欄位嗎？比如說晶片的資訊？

**【A】** 嗯，金片，實務上會啦，不過這邊你就用 wafer ID 代表它的 ID 就好了。對，我想一下，如果你要讓它豐富一點，你可以再加一個叫做**晶圓尺寸**。晶圓尺寸就是 300mm、200mm，就這兩個去設計。

## 四、機台與 Recipe 管理

**【Q】** recipe 是否可理解為由實驗室預先建立的流程與參數模板？某些 recipe 是否只能搭配特定機台使用？

**【A】** 嗯，你的理解是對的。首先機台的話，你要去管理它的機台能力嘛，它可以執行哪些實驗。那這個機台要執行這一種實驗就會有它的那個 recipe。所以你就是可能說這個 recipe 搭配這一組實驗、搭配這一組實驗是怎樣的。就是說這個機台要完成這個實驗項目，它可以用的 recipe 是什麼，類似這樣去規劃他們的關係。

**【Q】** 廠區使用者是否只能選用既有 recipe，而不能自行新增 recipe？廠區使用者選定 recipe 後，是否可以調整其中部分參數？

**【A】** 沒有沒有沒有。廠區使用者他想要的是委託實驗室幫他做一些實驗，就這樣而已。他把東西送過去之後什麼事都不知道，他就等結果。那要怎麼做實驗、用什麼機器、然後用什麼 recipe 怎麼做，就是 lab 實驗室的人的職責。

所以廠區使用者最多就是說，我跟實驗室問了一些，你們這個實驗室有很多實驗室，所以他可能會想說這個 RA 實驗室（可靠度分析實驗室），它可以幫我做比如說高溫烘烤測試好了。那我就會開張委託單給 RA 實驗室說，我有十片 wafer 幫我做高溫烘烤測試，烤個 300 小時，這樣子的委託單。然後就送給 Lab 去幫他做實驗這樣。

他可能會選或是說他想要某一個實驗項目，但他不會、他不太 care 這個實驗項目是怎麼被執行的，有執行完就好。

**【Q】** 所以廠區使用者是從實驗項目的角度去委託，而不是從 recipe 的角度？

**【A】** 對。你可以有一個實驗項目管理表讓實驗室的人可以去新增嘛，那廠區使用者就是用現有的項目去選那個他可以執行的東西。角度有點奇怪啦，就是廠區的人應該會從實驗項目那邊去委託。然後實驗室的人、lab 的人才是看這個實驗項目來決定要放到哪一個機台、用什麼參數嘛。

**【Q】** 所以 recipe 是實驗室內部的東西，廠區使用者不會接觸到？

**【A】** 對，recipe 這一層就太細了，它是實驗室內部的 domain。

**【Q】** 可以問一下有沒有實驗內容的例子，然後跟相對應會用到怎麼樣的 recipe 嗎？

**【A】** 實驗內容的例子喔，你可以 Google 一下**半導體品質測試**。那應該會有蠻多跟半導體品質相關的實驗室，比如說 RA、MA、FA、TM，就是這種在半導體業界一定都會有的實驗室。那公司就蠻多的啊，所以你應該可以 Google 項目啊，什麼電性測試啊、高溫烘烤，類似這種。

## 五、委託單需求

**【Q】** 委託單送出後，在簽核前廠區使用者是否仍可修改？

**【A】** 還沒簽核之前，我想一下喔。送出就不行。因為他都送出了嘛，那如果你要設計的話，你可以讓他在送出前有一個**草稿的狀態**嘛。因為送出就會進入簽核的流程，那時候改的話就會讓幫你簽核的人很困擾。

**【Q】** 那如果真的填錯了怎麼辦？

**【A】** 對，所以實際上實務上就是如果你真的覺得寫錯了，你就聯絡簽核的人，然後請他幫你**退單**重寫這樣。

**【Q】** 簽核結果是否應包含核准、退回、拒絕？退回和拒絕的差別？

**【A】** 有可能退回。拒絕理由都可以啦，就是你要讓主管可以留個 comment 嘛，拒絕的 reason 這樣。

**【Q】** 被退回後的委託單，是回到原本的狀態還是變成另一個狀態？

**【A】** 你說被退回去嗎？這個就你們設計。不過通常會是另一個狀態，因為嗯，我想一下喔。這個基本上可以自己發揮啦，因為沒有寫這麼細。

## 六、WIP 與分貨

**【Q】** 題目中的 WIP，是否可理解為樣品經分貨後形成的中間追蹤單位？

**【A】** 不過實際上一個 WIP 會等於一件事情。就是說這個動作，在這一——可能是一到多個樣品，然後在這個機台上發生的事情。

**【Q】** 所以 WIP 和樣品的關係是什麼？是從樣品的角度看還是從 WIP 的角度看？

**【A】** 對，所以可能你可以想一下從 WIP 的角度來看、什麼時候會從樣品的角度來看。然後去，你應該是想要做看需不需要做報表。

實務上啦，對實驗室在做那個下貨，我們會稱為——我們那個權責上是製造部，**製造部的人會關注的是 WIP 的單位**。然後他會去看說這個機台可能在單位時間內、一周內多執行了多少個 WIP，這種感覺。

**對啊，樣品通常會是開單的人在意的維度。** 因為他其實不管你怎麼下貨嘛，他只是在意他自己手上的東西有沒有做完實驗。

**【Q】** 一個樣品是否可分成多個 WIP？還是反過來多個樣品合成一個 WIP？

**【A】** 不會。你把它想成**樣品是實體**嘛，然後 **WIP 是虛擬的帳**。那一個 WIP 它就是一到多個實體。所以你可以把三片 wafer、三個樣品綁在一起一起進機，就是把它綁成一個 WIP。然後以這個 WIP 的單位，這三片一起進去機器裡面一起出來。

**【Q】** 那出來之後，這三片還是同一個 WIP 嗎？

**【A】** 不一定，只有看流程的複雜度啊。因為可能有這三片只是好可以合在一起進機，它下面可能各自要去走不同的流程嘛。所以接下來可能會再拆批，2-1 或是 1-1-1，然後去分別做其他事情。

**【Q】** 那拆批的部分會不會很複雜？

**【A】** 那邊就有點複雜了。沒有要求。實際上是這樣沒錯啦。對啊，因為這次的題目沒有要設計 flow（流程），沒有要設計流程。像台積一個製程三四千個站點再跳再走。

## 七、機台 Capacity 與 WIP 組合

**【Q】** 那機台的 capacity 要怎麼設計？

**【A】** 對，上就是你在設計機台這個東西的時候，我說機台的 model，你可能需要設定它的 **capacity** 嘛。它可以最多處理多少片晶圓。那你是不是就可以把各個委託單需要進機的合在一起成為一個 WIP，然後一起進去，發揮這個機台最大的效能。

## 八、派貨

**【Q】** 派貨是否需要模擬派貨過程，還是派貨後一定會到達？

**【A】** 直接到就好。喔，你們想得很細喔。實際上是會（沒到的），這邊就用**派貨就代表他一定會到**。應該說你們可以先把派就當做他已經開始做了，派下去做了，這樣子來講。

**【Q】** 所以派貨就等於開始做實驗？

**【A】** 對，派上去，然後他就開始做實驗。就用這個來出發就好。雖然以公司來說，派到派過去到實際上開始可以做，中間還有很多故事。複雜是一件事啊，然後就是派系統還有那個物流的管理。對，所以它可能 wafer 要走很遠，那走很遠就會有跨站跨區的問題。然後還要排隊，前面是不是那個機台已經有東西在執行了，所以你可能還要在某一個地方先存下來等它做完。

---

## 九、Recipe 管理權限

**【Q】** 那 recipe 的管理是誰負責？

**【A】** 實務上對 recipe 的管理就是實驗室的人。所以題目說要提供 recipe 的管理功能，就是要給實驗室的人在操作系統的時候可以去**新增、修改、刪除、查詢**，這種感覺。

## 十、異常處理

**【Q】** 廠區使用者需不需要知道實驗過程中的異常？

**【A】** 應該不用喔。因為對廠區的人來說，他其實沒有很在意你那邊是不是有發生意外嘛。所以應該是開始做實驗之後，以廠區的人來說，他就只看完成了沒。

**【Q】** 那如果異常沒辦法完成怎麼辦？

**【A】** 想得很細喔。因為異常沒辦法完成，通常實驗室就要去求助，可能別的公司能不能幫他們完成。嗯，你可以設計一個**中止**的狀態啊。就是因為任何原因就停掉了。因為有時候是 lab 實驗室執行到一半，反而是開單的人不想做了。然後可能他就一通電話打過來：「誒你那一個單號 XX 就幫我把它停掉、停掉。」類似這種感覺。

---

## 十一、自動結單

**【Q】** 機台可不可以自動接單跟自動結單？

**【A】** 那那條路我會說，如果要讓機台可以自動接單，代表你要有機台可以自己結單的能力嘛。就是它到底什麼時候會結、什麼時候會完成、怎樣才算完成。

首先你可以先分成，我們說機台自動化。如果沒有機台自動化的能力的時候，實驗室的人會怎麼使用這一套系統？就是做完實驗，然後打開系統，然後說這個實驗已經做完了，可能有一個完成的按鈕，然後點下去可能會輸入一些實驗數據嘛。那這個就是**手動作帳**。

那如果你設計的機台是有自動化能力的話，那它可能會過一段時間，然後傳一些數據到你的伺服器，然後就觸發**自動關單**，就類似這種感覺。可以這樣設計。

**【Q】** 樣品的 entity 是什麼時候建立的？是委託單送出的時候就建立嗎？

**【A】** 可以自己設計啊，沒有講這麼細。不過我們的做法是**送單的時候，樣品的那個 entity 就出來了**。因為要記錄它的狀態嘛。

## 十三、接樣異常

**【Q】** 接樣的時候會不會有異常？比如說樣品沒送到？

**【A】** 沒有啊，一定會到，但可能**料不符**啊。他送來的東西跟開單的時候申請填的東西不一樣，你就要拒絕嘛。

## 十四、WIP 與樣品的異常狀態關係

**【Q】** 什麼時候會發生異常？是在實驗的過程嗎？機台異常的話，WIP 和樣品的狀態怎麼處理？

**【A】** 機台異常的話，**WIP 應該要有異常的狀態**，因為 WIP 就是一起進的單位嘛。那**樣品的狀態我覺得就無關了**。那你也可以把他們掛鉤在一起啦，因為樣品的狀態你也可以說它都是在「實驗中」嘛。那實驗中的細節應該是 WIP 的狀態再描述。

## 訪談關鍵結論摘要

| 項目 | 結論 |
|------|------|
| 角色 | 廠區使用者（多人）、實驗室人員、實驗室主管，共三種角色 |
| 送件 vs 送出 | 「送出」= 委託單在系統上送出；「送件」= 實體樣品送到實驗室 |
| 欄位設計 | 自行設計，合理即可。樣品用 wafer ID，可加晶圓尺寸（300mm/200mm） |
| 委託單修改 | 送出後不可修改，送出前可有草稿狀態；退回後可修改重送 |
| 簽核 | 需有退回/拒絕機制，主管可留 comment 說明原因 |
| 廠區使用者權限 | 只能選擇實驗項目委託，不接觸 recipe 和機台細節 |
| Recipe 管理 | 實驗室人員負責 CRUD，與機台和實驗項目有對應關係 |
| WIP 定義 | 虛擬帳，一個 WIP = 一到多個樣品在同一機台上的一次操作 |
| WIP vs 樣品視角 | 樣品 = 開單者關心；WIP = 製造部/實驗室關心 |
| 機台 Capacity | 機台需設定最大處理片數，可將多個樣品合併為一個 WIP 進機 |
| 派貨 | 簡化設計：派貨 = 開始做實驗，不需模擬物流過程 |
| 異常處理 | WIP 層級處理異常，樣品狀態可維持「實驗中」；需有中止狀態 |
| 自動結單 | 可設計手動作帳 + 機台自動化傳數據觸發自動關單兩種模式 |
| 樣品建立時機 | 委託單送出時就建立樣品 entity |
| 接樣異常 | 樣品一定會到，但可能料不符需拒絕 |
| 流程複雜度 | 本次不需設計完整 flow，不需拆批等複雜邏輯 |
