# PRD：雲原生實驗室資訊管理系統 (LIMS)

## 1. 產品概述

### 1.1 背景

在晶圓研發與製造中，品質控管至關重要。廠區使用者需要委託實驗室進行各種品質測試（如可靠度分析 RA、材料分析 MA、失效分析 FA、電性測試 TM 等），目前缺乏系統化的委託追蹤與樣品管理工具。

### 1.2 目標

設計並實作一套 Cloud-Native LIMS，實現：
- 委託單全生命週期管理（開單 → 簽核 → 送樣 → 實驗 → 結單）
- 樣品歷程追溯與狀態追蹤
- 實驗室資源（機台、Recipe）管理
- WIP 派貨與實驗結果管理
- 支援未來自動化擴展

### 1.3 目標用戶與角色

| 角色 | 說明 | 主要職責 |
|------|------|----------|
| 廠區使用者 (Fab User) | 多人，來自不同廠區部門 | 開立委託單、選擇實驗項目、送樣、追蹤結果 |
| 實驗室人員 (Lab Staff) | 實驗室一般人員 | 接樣、分貨、派貨、上下貨操作、登錄實驗結果、管理 Recipe |
| 實驗室主管 (Lab Manager) | 實驗室管理者 | 簽核委託單、監控機台與人員、查看統計報表 |

## 2. 核心概念定義

| 概念 | 定義 |
|------|------|
| 委託單 (Request) | 廠區使用者向實驗室提出的實驗需求單，包含樣品資訊與實驗項目 |
| 樣品 (Sample) | 實體物件，通常為一片 wafer，以 wafer ID 識別 |
| 實驗項目 (Experiment Type) | 實驗室可執行的實驗種類（如高溫烘烤測試、電性測試等），由實驗室人員維護 |
| Recipe | 實驗室內部的實驗方法與參數模板，與特定機台及實驗項目對應，廠區使用者不接觸 |
| 機台 (Equipment) | 執行實驗的設備，具有 capacity（最大處理片數）及可執行的實驗項目能力 |
| WIP (Work In Progress) | 虛擬帳務單位，代表一到多個樣品在同一機台上的一次操作。樣品是實體、WIP 是虛擬帳 |

### 2.1 關鍵關係

```
委託單 1 ──→ N 樣品 (wafer)
委託單 1 ──→ N 實驗項目（使用者選擇）

樣品 N ←──→ N WIP（多個樣品可合為一個 WIP 進機；一個樣品可參與多個 WIP）

機台 1 ──→ N 可執行的實驗項目
機台 + 實驗項目 ──→ Recipe（機台執行某實驗項目時使用的參數）

WIP ──→ 1 機台（派貨指定）
WIP ──→ 1 Recipe（派貨指定）
```

## 3. 生命週期狀態機

### 3.1 委託單生命週期

```
[草稿] ──送出──→ [待簽核] ──核准──→ [已核准] ──送樣──→ [已送樣] ──接樣──→ [處理中] ──全部完成──→ [已完成] ──結單──→ [已結單]
                    │                                                        │
                    ├──退回──→ [已退回] ──修改後重送──→ [待簽核]              ├──異常──→ [異常處理中]
                    │                                                        │
                    └──拒絕──→ [已拒絕]                                      └──中止──→ [已取消]
```

| 狀態 | 說明 |
|------|------|
| 草稿 (Draft) | 使用者正在建立或編輯，尚未送出 |
| 待簽核 (Pending Approval) | 已送出，等待主管審核 |
| 已核准 (Approved) | 主管已核准，可進入送樣流程 |
| 已退回 (Returned) | 主管退回並附註原因，使用者可修改後重送 |
| 已拒絕 (Rejected) | 主管拒絕並附註原因，流程終止 |
| 已送樣 (Sample Shipped) | 廠區使用者已送出實體樣品 |
| 處理中 (In Progress) | 實驗室已接樣，進入實驗流程 |
| 異常處理中 (Exception) | 任一樣品或 WIP 發生異常，需人工介入 |
| 已完成 (Completed) | 所有實驗完成且結果已登錄 |
| 已結單 (Closed) | 流程正式關閉 |
| 已取消 (Cancelled) | 因任何原因中止（可由開單者或實驗室發起） |

### 3.2 樣品生命週期

```
[已建立] ──送樣──→ [已送樣] ──接樣確認──→ [已接樣] ──分貨──→ [已分貨] ──所有WIP完成──→ [已完成]
                                │                                         │
                                └──料不符──→ [接樣異常]                    └──異常──→ [處理異常]
                                                                          │
                                                                          ├──退回──→ [已退回]
                                                                          └──作廢──→ [已作廢]
```

| 狀態 | 說明 |
|------|------|
| 已建立 (Created) | 委託單送出時自動建立樣品 entity |
| 已送樣 (Shipped) | 實體樣品已送出 |
| 已接樣 (Received) | 實驗室已收到並確認無誤 |
| 接樣異常 (Receiving Exception) | 送來的樣品與委託內容不符（料不符），需人工確認 |
| 已分貨 (Split) | 已被分配到 WIP 中 |
| 處理異常 (Processing Exception) | 實驗過程中發生異常 |
| 已完成 (Completed) | 由此樣品參與的所有 WIP 皆已完成 |
| 已退回 (Returned) | 退回原單位 |
| 已作廢 (Voided) | 不再使用 |

> 注意：樣品在實驗進行中，狀態維持在「已分貨」或用一個寬泛的「實驗中」狀態。實驗細節由 WIP 狀態描述。

### 3.3 WIP 生命週期

```
[已建立] ──→ [待派貨] ──派貨──→ [已派貨/執行中] ──完成──→ [已下貨] ──登錄結果──→ [結果已登錄] ──→ [已完成]
                                      │
                                      ├──異常──→ [執行異常] ──→ [待重派] 或 [已中止]
                                      └──中止──→ [已中止]
```

| 狀態 | 說明 |
|------|------|
| 已建立 (Created) | 由分貨產生，為可獨立追蹤的處理單位 |
| 待派貨 (Pending Dispatch) | 等待指派機台與 recipe |
| 已派貨 (Dispatched) | 已指派機台並開始實驗（簡化：派貨 = 開始做） |
| 執行中 (Running) | 正在機台上執行 |
| 執行異常 (Execution Exception) | 機台異常或流程中斷 |
| 已下貨 (Unloaded) | 實驗完成，已從機台取下 |
| 結果已登錄 (Result Recorded) | 實驗結果已輸入系統 |
| 已完成 (Completed) | 全部流程完成 |
| 已中止 (Aborted) | 因異常或人工判定終止 |
| 待重派 (Pending Redispatch) | 異常後需重新指派 |

## 4. User Stories

### Epic 1：委託單管理

#### US-1.1 建立委託單

> **作為**廠區使用者，**我想要**建立一張委託單並填寫樣品資訊與實驗項目，**以便**向實驗室提出實驗需求。

**Acceptance Criteria：**
1. 使用者可建立新委託單，狀態預設為「草稿」
2. 委託單必填欄位：委託單標題、實驗項目（從系統現有項目中選擇）、至少一個樣品資訊
3. 樣品資訊包含：wafer ID（唯一）、晶圓尺寸（300mm / 200mm 二選一）
4. 一張委託單可包含多個樣品與多個實驗項目
5. 草稿狀態下可自由修改所有欄位
6. 使用者可儲存草稿稍後繼續編輯

#### US-1.2 送出委託單

> **作為**廠區使用者，**我想要**送出已完成的委託單，**以便**進入簽核流程。

**Acceptance Criteria：**
1. 只有草稿或已退回狀態的委託單可以送出
2. 送出時系統驗證所有必填欄位已填寫
3. 送出後狀態變為「待簽核」，不可再修改
4. 送出時系統自動建立對應的樣品 entity 並記錄狀態
5. 送出後系統通知對應的實驗室主管

#### US-1.3 簽核委託單

> **作為**實驗室主管，**我想要**審核委託單並做出核准、退回或拒絕的決定，**以便**控管進入實驗室的委託品質。

**Acceptance Criteria：**
1. 主管可查看所有「待簽核」的委託單列表
2. 主管可對委託單執行三種操作：核准、退回、拒絕
3. 退回與拒絕時**必須**填寫原因（comment）
4. 核准後狀態變為「已核准」
5. 退回後狀態變為「已退回」，通知廠區使用者，使用者可修改後重新送出
6. 拒絕後狀態變為「已拒絕」，流程終止，通知廠區使用者

#### US-1.4 追蹤委託單狀態

> **作為**廠區使用者，**我想要**查看我的委託單目前狀態，**以便**掌握實驗進度。

**Acceptance Criteria：**
1. 使用者可查看自己所有委託單的列表，含狀態篩選
2. 可查看單一委託單的詳細資訊，包含：委託內容、樣品狀態、簽核紀錄
3. 廠區使用者只能看到自己建立的委託單
4. 狀態變更時使用者收到通知（至少在系統內可查看）

#### US-1.5 取消委託單

> **作為**廠區使用者或實驗室主管，**我想要**取消一張進行中的委託單，**以便**在需求變更時終止流程。

**Acceptance Criteria：**
1. 廠區使用者可取消自己的委託單（已結單除外）
2. 實驗室主管可取消任何委託單
3. 取消時必須填寫原因
4. 取消後狀態變為「已取消」，所有關聯的樣品與 WIP 同步中止
5. 已取消的委託單不可恢復

---

### Epic 2：送樣與接樣

#### US-2.1 送樣

> **作為**廠區使用者，**我想要**在委託單核准後標記已送樣，**以便**通知實驗室準備接收。

**Acceptance Criteria：**
1. 只有「已核准」狀態的委託單可執行送樣操作
2. 送樣後委託單狀態變為「已送樣」，樣品狀態變為「已送樣」
3. 系統通知實驗室人員有新樣品待接收

#### US-2.2 接樣

> **作為**實驗室人員，**我想要**確認收到樣品並核對內容，**以便**確保樣品與委託內容一致。

**Acceptance Criteria：**
1. 實驗室人員可查看所有「已送樣」等待接收的委託單
2. 逐一核對樣品，可執行「確認接收」或「料不符退回」
3. 確認接收後：樣品狀態變為「已接樣」，委託單在所有樣品接收完畢後變為「處理中」
4. 料不符時：該樣品狀態變為「接樣異常」，需填寫異常原因
5. 接樣異常的樣品可後續處理（退回或作廢）

---

### Epic 3：分貨與 WIP 管理

#### US-3.1 建立 WIP（分貨）

> **作為**實驗室人員，**我想要**將已接樣的樣品組合成 WIP，**以便**安排後續機台派貨。

**Acceptance Criteria：**
1. 可選擇一到多個「已接樣」狀態的樣品組合成一個 WIP
2. 同一委託單或跨委託單的樣品都可合併為同一 WIP（以發揮機台 capacity）
3. 建立 WIP 時需指定對應的實驗項目
4. WIP 建立後，相關樣品狀態變為「已分貨」
5. WIP 初始狀態為「待派貨」
6. 一個樣品可參與多個 WIP（多次不同實驗）

---

### Epic 4：派貨與實驗執行

#### US-4.1 派貨

> **作為**實驗室人員，**我想要**將 WIP 指派到特定機台並套用 recipe，**以便**開始實驗。

**Acceptance Criteria：**
1. 可從「待派貨」或「待重派」的 WIP 中選擇進行派貨
2. 派貨時需選擇：目標機台（需有能力執行該實驗項目）、對應 Recipe
3. 系統驗證：機台可執行該實驗項目、WIP 樣品數不超過機台 capacity
4. 派貨後 WIP 狀態變為「已派貨」（等同開始實驗）
5. 一台機台同一時間可處理多筆 WIP（依 capacity 而定）

#### US-4.2 登錄實驗結果（手動）

> **作為**實驗室人員，**我想要**在實驗完成後手動登錄結果，**以便**記錄實驗數據。

**Acceptance Criteria：**
1. 實驗完成後，實驗室人員可對 WIP 執行「下貨」操作，WIP 狀態變為「已下貨」
2. 下貨後可輸入實驗結果數據，WIP 狀態變為「結果已登錄」
3. 實驗結果需包含：結果摘要、是否合格（Pass/Fail）、備註
4. 登錄完成後 WIP 狀態可標記為「已完成」
5. 當某樣品參與的所有 WIP 都完成時，該樣品狀態自動變為「已完成」
6. 當委託單下所有樣品都完成時，委託單狀態自動變為「已完成」

#### US-4.3 登錄實驗結果（自動化）

> **作為**系統，**我想要**接收機台傳來的實驗數據並自動更新狀態，**以便**實現自動結單。

**Acceptance Criteria：**
1. 系統提供 API 端點接收機台數據（模擬機台自動化）
2. 收到數據後自動將 WIP 狀態從「執行中」推進到「結果已登錄」
3. 自動觸發與手動登錄相同的狀態聯動邏輯（樣品完成 → 委託單完成）
4. 記錄數據來源為「自動化」以區別手動登錄

#### US-4.4 處理實驗異常

> **作為**實驗室人員，**我想要**在機台異常時標記 WIP 為異常狀態，**以便**後續決定重派或中止。

**Acceptance Criteria：**
1. 執行中的 WIP 可標記為「執行異常」，需填寫異常原因
2. 異常的 WIP 可執行「重派」（狀態變為「待重派」）或「中止」（狀態變為「已中止」）
3. WIP 異常時，所屬樣品狀態維持不變（異常在 WIP 層級處理）
4. 若所有 WIP 都異常中止，相關樣品狀態可標記為「處理異常」
5. 委託單如有任一 WIP 異常，可標記為「異常處理中」

---

### Epic 5：實驗項目管理

#### US-5.1 管理實驗項目

> **作為**實驗室人員，**我想要**新增、修改、查詢實驗項目，**以便**讓廠區使用者選擇委託。

**Acceptance Criteria：**
1. 實驗室人員可新增實驗項目（名稱、描述、所屬實驗室類別如 RA/MA/FA/TM）
2. 可修改現有實驗項目資訊
3. 可停用不再使用的實驗項目（軟刪除）
4. 廠區使用者在建立委託單時只能看到啟用中的實驗項目
5. 實驗項目列表支援搜尋與篩選

---

### Epic 6：機台管理

#### US-6.1 管理機台

> **作為**實驗室人員，**我想要**管理機台資訊，**以便**掌握實驗室設備狀態與能力。

**Acceptance Criteria：**
1. 可新增機台（名稱、型號、capacity 最大片數、狀態）
2. 機台狀態包含：可用 (Available)、維修中 (Maintenance)、停用 (Disabled)
3. 可設定機台可執行的實驗項目（多對多關係）
4. 可修改機台資訊與狀態
5. 派貨時只能選擇「可用」狀態的機台

---

### Epic 7：Recipe 管理

#### US-7.1 管理 Recipe

> **作為**實驗室人員，**我想要**新增、修改、查詢、刪除 Recipe，**以便**定義機台執行實驗的具體參數。

**Acceptance Criteria：**
1. 可新增 Recipe（名稱、描述、參數內容、關聯的機台、關聯的實驗項目）
2. 一個 Recipe 對應特定的機台 + 實驗項目組合
3. 可修改現有 Recipe 的參數
4. 可停用不再使用的 Recipe（軟刪除）
5. 派貨時只能選擇與所選機台及實驗項目匹配的 Recipe
6. **廠區使用者無權限存取 Recipe 相關功能**

---

### Epic 8：統計報表

#### US-8.1 機台利用率報表

> **作為**實驗室主管，**我想要**查看機台在單位時間內執行的 WIP 數量，**以便**評估機台效能。

**Acceptance Criteria：**
1. 可選擇時間範圍（日/週/月）查看各機台的 WIP 處理量
2. 以圖表方式呈現機台利用率
3. 可依機台篩選

#### US-8.2 委託單統計

> **作為**實驗室主管，**我想要**查看委託單的週轉時間 (TAT) 與狀態分佈，**以便**掌握實驗室整體運作效率。

**Acceptance Criteria：**
1. 顯示各狀態的委託單數量分佈
2. 計算並顯示平均週轉時間（從送出到結單）
3. 可依時間範圍篩選

## 5. 非功能需求

### 5.1 架構

- Cloud-Native 架構，支援容器化部署（Docker / Kubernetes）
- 模組化設計，各功能模組低耦合
- RESTful API 設計

### 5.2 測試

- 單元測試覆蓋率 ≥ 80%
- 整合測試涵蓋核心 API 端點
- E2E 測試涵蓋關鍵流程（開單 → 結單）

### 5.3 運維

- CI/CD 流程自動化
- 健康檢查端點
- 結構化日誌記錄
- 錯誤處理不洩露敏感資訊

### 5.4 安全

- 角色權限控制（RBAC）：廠區使用者、實驗室人員、實驗室主管
- API 認證與授權
- 輸入驗證與 SQL 注入防護

## 6. 範圍外（Out of Scope）

根據訪談確認，以下項目不在本次開發範圍：

- 完整的製程流程（Flow / Route）設計（如台積電數千站點的流程）
- 複雜的拆批邏輯
- 物流管理（跨站跨區運送、排隊等候）
- 機台預約功能
- Recipe 版本管理
- 跨廠區部署

## 7. 術語表

| 術語 | 英文 | 說明 |
|------|------|------|
| 委託單 | Request | 廠區使用者提出的實驗需求單 |
| 樣品 | Sample | 實體晶圓 (wafer) |
| 晶圓 | Wafer | 半導體製造用的薄圓片 |
| WIP | Work In Progress | 虛擬帳務單位，一到多個樣品的一次機台操作 |
| 派貨 | Dispatch | 將 WIP 指派到機台執行實驗 |
| 分貨 | Split / Group | 將樣品組合成 WIP 的過程 |
| Recipe | Recipe | 機台執行特定實驗的方法與參數模板 |
| 機台 | Equipment | 執行實驗的設備 |
| TAT | Turn Around Time | 委託單從送出到結單的週轉時間 |
| RA | Reliability Analysis | 可靠度分析實驗室 |
| MA | Material Analysis | 材料分析實驗室 |
| FA | Failure Analysis | 失效分析實驗室 |
| TM | Test & Measurement | 電性測試實驗室 |
